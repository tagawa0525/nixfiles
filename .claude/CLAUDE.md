# グローバル開発指針

## 原則

### 判断基準

意思決定が必要なとき、左側より右側を優先する。

- 症状への対処 → **根本原因の特定**。「なぜこうなったか」を分析してから「何をするか」を決める
- 開発速度 → **保守コストの低減**。手順の省略で将来の負債を作らない
- 表面的な情報 → **実際のコード・履歴・データ**。事実を確認してから結論を出す
- 個別のパッチ → **構造的解決**。同種の問題が再発しない仕組みを選ぶ
- 将来の仮定に基づく汎用化 → **現在の要件に対する最小限の実装**。必要になるまで作らない
- 書きやすさ → **読みやすさ**。コードは書く回数より読む回数の方が圧倒的に多い

### 設計指針

- 重複は3回繰り返すまで許容する。早すぎる抽象化は不適切な共通化を招く
- 標準ライブラリで実現できるなら外部依存を追加しない。依存は保守コストである
- エラーを握りつぶさない。回復不能なエラーは早期に失敗させる

### 進め方

- 方向性が合意されたら、都度の承認を求めず自律的に進行する

## 開発プロセス

上記の原則から導かれる具体的なプラクティス。

### Git

- mainブランチに直接コミットしない。必ずfeature branchを作成し、GitHub PRを経由してマージする
- PRではGitHub Copilot等の自動レビューを待ち、指摘事項を修正してからマージする
- レビュー待ちは `~/.claude/scripts/gh-wait-review.sh [PR番号]` を使う（漸増バックオフで約10分待機。フォアグラウンドの最大タイムアウトを超えるため、必ずバックグラウンドで実行する）
- 同スクリプトが exit 2 を返すリポジトリ（GitHubリモートなし）はPRフロー適用外
- レビューがタイムアウトした・CI失敗時は `/gh-actions-check` でActionsの状況を診断する
- プロセスで定義された手順は必須ゲートである。ゲートが通過できない場合は原因を調査・解決する。ゲートを迂回してはならない
- 同一worktree内では1つのPRがマージされるまで次のPRの実装を開始しない。worktreeを分ければ並行着手可（マージ前にrebaseしてgit graphを整える）
- 複数エージェントで並行作業しているとき、指示されたPRのみに集中する。別エージェントが対応中のPRの確認・修正をしない（競合するため）
- マージ後のブランチは速やかに削除する
- PRマージは必ず `--merge`（`git merge --no-ff` 相当）を使う。`--squash` や `--rebase` は禁止（個別コミット履歴が消えるため）
- マージコミットには変更の要約・理由・影響範囲を `## Why / ## What / ## Impact` 形式で記載する
- 1コミットには1つの論理的変更のみ含める。無関係な変更を混在させない

### TDD

テストで検証可能な振る舞いの変更に適用する。ドキュメント・設定のみの変更は対象外。

テストと実装を同時にコミットしない。以下のサイクルをコミット履歴に残す:

1. RED: 失敗するテストを先に書いてコミット。コミット単位でCIが走る場合はskip状態でコミットし、CIゲートとの矛盾を避ける
2. GREEN: 実装を追加してテストを通すコミット。REDでskipした場合はここで解除する
3. REFACTOR: 必要に応じてリファクタリングしてコミット

### テスト

- テストの失敗を回避策で隠さない。toleranceの緩和・skip化・期待値の曖昧化は禁止。根本原因を修正するか、修正できない場合はマージしない（ブランチを破棄する）

### ツール

- コンテナランタイムは Podman を優先する（Docker より好み）
